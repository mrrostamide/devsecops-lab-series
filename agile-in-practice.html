Coding habits
▾

Below are two coding practices shown through situations where their absence caused a real problem for StackSafe's team.

Recall: XP = Scrum's team rhythm + these coding habits, used together, not instead of each other
A
Pair programming: spot the bug
James wrote the payment fee calculation alone over two days. Here are the steps his code follows. Click the step you think causes a problem.
1.Take the amount the user wants to send, in their own currency.
2.Calculate the fee as 2% of that amount, in the user's own currency.
3.Convert the full amount into the receiver's currency, using today's exchange rate.
4.Subtract the fee from step 2 from the converted amount from step 3.
5.Send the final amount to the receiver.
B
Refactoring
A developer who wrote StackSafe's payment module left the company. When Amir, the Scrum Master, asked the remaining team to explain a section of that code, nobody could quickly say what it did. Here is a simplified version of that code.
BEFORE
function processPayment(amount):  if user.loggedIn == false: return error  fee = amount * 0.02  converted = amount * exchangeRate  result = converted - fee  display(result)  sendEmail(user.email, result)
Every step is in one unlabelled block, and the number 0.02 appears with no explanation of what it means.
AFTER
SERVICE_FEE_PERCENT = 2% function processPayment(amount):  checkUserLoggedIn(user)  fee = calculateFee(amount, SERVICE_FEE_PERCENT)  converted = convertCurrency(amount, exchangeRate)  result = converted - fee  showResult(result)  sendConfirmationEmail(user)
The number 0.02 became a named value: SERVICE_FEE_PERCENT.
One flat block became a sequence of clearly named steps.
Nothing this returns to the user has changed. Only how the code is organised has changed.
Discussion question
StackSafe is now running Scrum. The engineering lead also wants the team to adopt pair programming and regular refactoring, like you just saw. One developer says: "We already follow Scrum, why change how we code too?" How would you respond?
