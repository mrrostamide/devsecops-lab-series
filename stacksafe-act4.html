Act 4 · The story

The Pull Request

In Act 2, Priya merged her branch into her own copy of StackSafe. That made the feature real, but only in her fork. The real, shared project has never seen it. It still doesn't know Priya's branch exists.

A Pull Request closes that gap. It isn't a second merge of the same thing, it's the first time this change is proposed to the actual team project. And this time, someone looks at it before it's accepted.

What is a Pull Request? A Pull Request (often shortened to "PR") is a formal request asking a project's owner to accept your changes into their repository. It shows exactly what you want to add or change, side by side with what's already there, and lets someone review it before anything is actually merged. Unlike the merge you did in Act 2, which happened instantly with nobody checking it, a Pull Request always has a pause built in.

Resume your Codespace

Same as before: go to github.com/codespaces and open your existing StackSafe codespace, or create one if needed.

Make sure you're not behind

Before proposing anything, check whether the real project has changed since you forked it.

git fetch upstream

Plain words: fetch only looks. It checks whether upstream has commits you don't have yet, without changing any of your own files. Always safe to run.

Take a look at what's actually different before bringing anything in:

git log main..upstream/main
Predict: if upstream has a new commit you don't have, what should you do before opening a PR?
🔒 Answer the question above to continue.
git pull upstream main --no-rebase --no-edit

Plain words: You may see a message about "divergent branches", that's expected here, not an error. Your local main has Alex's and Priya's work upstream doesn't have, and upstream now has a new file yours doesn't have. Since neither side is simply "ahead" of the other, Git needs to combine them into a real merge, rather than just fast-forwarding. The --no-rebase --no-edit part tells Git exactly how to do that (a normal merge) without opening a text editor to ask you anything.

Open CONTRIBUTING.md in your file explorer to confirm it's really there.

Make sure your fork is fully up to date

git push origin main

Plain words: This sends the merge commit from Step 1, the one that pulled in CONTRIBUTING.md, up to your fork. Your fork's main is now the most current version of everything you've built so far.

Open the Pull Request

  1. Go to your fork's main page on GitHub
  2. Click the "Contribute" dropdown, near the top of the page, next to "Sync fork"
  3. Click "Open pull request"

This takes you to a single page with everything you need: a comparison bar at the top, a title field, and a description box below it, all on one screen. Notice you're no longer looking at your own fork, you're now on the real StackSafe repository, since that's where this proposal actually lives.

Check the comparison bar first. Base repository is the real StackSafe project (upstream). Compare is your fork's main. A green "Able to merge" message means you're clear to continue.

Plain words: Notice this comparison includes everything you've done across the whole course so far, not just Priya's one line. That's a real side effect of how this course is structured, everything has happened on one shared main. In many real projects, PRs bundle several related commits together too. The lesson that matters here isn't a perfectly isolated diff, it's the review-then-merge workflow itself.

You may also see a note saying "this is your first time opening a pull request in this project", with a link to the contributing guidelines. That's the CONTRIBUTING.md file from Step 1, GitHub found it automatically and is pointing you to it, exactly what that file is for.

Predict: which direction is this Pull Request proposing to merge?
🔒 Answer the question above to continue.

Fill it in, then look at the diff

  1. Title: something short and specific, e.g. Add credit line from [Your Name], not just "changes" or "update"
  2. Description: since this PR includes everything from earlier Acts too, be specific about what you actually want reviewed, e.g.: "This includes the full project so far. The change I'd like reviewed is the credit line I added with my name, from an earlier exercise."

Scroll down. Below the description, you'll see the full list of commits being proposed, and further down, the actual file changes.

Plain words: This diff shows what's currently different between the real project and your fork, not a replay of every single commit. That's why some things from your history, like Sam's secret file, don't show up here at all, they were removed before this comparison, so there's nothing left to show. The commit list above the diff still records that they happened, if a reviewer wants to look closer, they can click into any individual commit.

Once you're satisfied it's correct, click "Create pull request."

Plain words: Nothing is merged yet. You've only proposed it. The real project's main branch is completely unaffected until someone accepts this.

Ask your instructor to review it

Once your PR is open, tell your instructor. They'll open it from their own repository's "Pull requests" tab (not yours, since it's proposing a change into their project) and leave a comment on the "Files changed" tab.

Plain words: In a real team, review isn't optional, and it isn't done by just anyone, it's usually done by whoever owns or maintains the project, or someone they've explicitly trusted with that responsibility. That's exactly what's happening here: your instructor is acting as the project's maintainer.

Only the maintainer can merge, and that's the actual lesson

Here's something worth noticing directly: if you look at your own PR, there's no working "Merge" button for you to click. That's not a bug, and it's not something you did wrong.

Plain words: Only the person who owns the repository (or someone they've given write access to) can merge a Pull Request into it. You, as a contributor working from a fork, can propose, but you can't approve your own proposal. This is one of the most important security ideas in this whole course: the person making a change and the person authorizing it are deliberately different people. That's exactly what real access controls and required reviewers enforce on real projects.

Your instructor will review and merge one representative PR live, as a class demonstration, showing the "Merged" badge appear in real time. Because everyone personalized their line back in Act 2, your instructor can actually merge everyone's contribution afterward, without them conflicting with each other.

If your specific PR isn't the one merged live, that's completely normal. In real open-source projects, most PRs are reviewed, but they aren't all merged in the same moment. What matters for this course is that you completed the proposal correctly.

Reference

Quick Glossary

Open this anytime, from any Act. Looking it up never affects your progress or locks.

Git

Saves versions of your project. Works offline.

GitHub

The website where Git projects are shared with a team.

Repository

A project folder Git is watching. Also called a "repo."

Codespaces

A ready-made computer in your browser. Nothing to install.

The terminal

Where you type commands and press Enter to run them.

Staging area

Where you collect exactly what you want to save next, using git add.

Commit

A permanent save point in Git's history. Made with git commit.

Fork

Your own personal copy of someone else's repository on GitHub.

Branch

A separate copy of the timeline, a safe space to try something without touching main.

Merge

Folding one branch's changes into another. Done with git merge.

.gitignore

Tells Git to never track certain files, but only from now on, not the past.

API key

A private password proving your program is allowed to use a service.

Rotate (a key)

Replacing an exposed credential with a new one. The real fix after a leak.
