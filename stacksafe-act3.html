Act 3 · The story

Sam's Mistake

A tiny shortcut. Almost every developer takes it at least once. This time, you'll see exactly what it costs.

Sam wants to add a small feature: a box that shows today's weather.

To get real weather data, the code needs a special password, called an API key. This key proves the request really comes from StackSafe.

"I'll just put the key in a file for now," Sam says. "I'll move it somewhere safer later."

Tap a card below to flip it and learn what each term means 👇

Sam takes a shortcut

Create weather_key.txt:

API_KEY=sk_test_12345abcdefg
git status
Predict: what does Git say about this file?
🔒 Answer the question above to continue.

Plain words: Git never automatically protects secrets, even one with an obvious name, that's entirely up to you.

git add weather_key.txt
git commit -m "Add API key for weather feature"

Before we push, look at the repo

Go to your StackSafe repository on github.com, look at the file list.

Plain words: This might look surprising, where's index.html? Where's Priya's team line? The truth is: everything you've built so far, in Act 1 and Act 2, has only ever existed inside your Codespace. You never pushed any of it.

Now push it:

git push origin main

Plain words: This is a bigger moment than it might look. This single push sends everything you've built so far to GitHub all at once. Up until this exact second, all of it was private to your Codespace. The instant this finishes, none of it is.

See it live

Go back to your StackSafe repository tab in the browser and refresh the page.

Click into weather_key.txt and open it.

Plain words: Compare this to the file list before, one push apart. This is now a real, public webpage. If this were a real key, anyone, or any bot scanning GitHub, could see it right now.

Sam sees it, and stops

Sam checks the repository online, just to confirm the push worked, and sees the key sitting there in plain text.

"Wait. That's a real-looking password, sitting on a public webpage. That doesn't feel right."

Plain words: This moment matters. In real teams, nobody double-checks every commit before it happens, mistakes like this slip through constantly. What you do after noticing is what actually matters.

The obvious fix, .gitignore

What is .gitignore? A special file that tells Git "never start tracking these files."
Predict: weather_key.txt was already committed and pushed. If Sam now adds it to .gitignore, will it disappear from today's file list on GitHub?
🔒 Answer the question above to continue.

How to create .gitignore:

  1. Right-click the StackSafe folder → New File
  2. Name it exactly .gitignore
  3. Inside, type: weather_key.txt
  4. Save
git add .gitignore
git commit -m "Add gitignore"
git push

Refresh your repository's file list on GitHub.

Yes, completely normal, not a risk. .gitignore is just a regular text file, like README.md, and almost every real public project has one, visible to everyone. There's nothing to hide about the file existing.

One small thing worth knowing: whatever you list inside .gitignore is also public. Listing weather_key.txt in it doesn't expose the secret itself, but it does quietly announce "a file with this name exists, or existed, here." A minor detail, not the main lesson of this Act, but good security awareness.

Sam tries one more thing, git rm --cached

Predict, part 1: this command tells Git "stop tracking this file, but leave it in your Codespace." Will weather_key.txt disappear from today's file list this time?
🔒 Answer the question above to continue.
git rm --cached weather_key.txt
git commit -m "Stop tracking weather_key.txt"
git push

Refresh the file list on GitHub.

Predict, part 2: does this also remove weather_key.txt from the commit that first added it, back in Step 1?
🔒 Answer the question above to see the proof.

How to review a commit on GitHub:

  1. Go to your repository's main page
  2. Click the commit count link near the top of the file list (e.g., "12 Commits")
  3. Find the commit "Stop tracking weather_key.txt", the one you just made
  4. Click it to open the diff. Red lines with a "-" are what was removed, and the old content is still fully readable right there

What actually fixes this

Deleting the file, or even rewriting history to remove it, doesn't fully solve this either, if it was ever pushed publicly, assume it's compromised. The only real fix is to rotate the key, generate a new one and invalidate the old one, regardless of what you do in Git. We'll cover cleaning secrets from history properly in Session 2, but the first and most important step is always: change the credential.

Do it securely, from the start

Sam gets a brand new, rotated key, weather_key_rotated.txt, and this time does it in the correct order.

First, open .gitignore (already exists from Step 5). Add a new line: weather_key_rotated.txt. Save.

Commit and push this .gitignore change, by itself, before the new key file exists at all:

git add .gitignore
git commit -m "Ignore weather_key_rotated.txt before it exists"
git push

Now, and only now, create weather_key_rotated.txt:

API_KEY=sk_live_98765zyxwv_rotated

Run:

git status

Plain words: Compare this to Step 1. Back then, git status showed weather_key.txt as untracked. This time, weather_key_rotated.txt doesn't appear anywhere in the output at all, because .gitignore already told Git to ignore it before the file existed. Git isn't just declining to track it, it's not even looking at it.

The one-sentence rule this Act teaches: .gitignore only protects a file if it's added before that file is ever created and committed. Prevention, done in the right order, is the only version of this that actually works.

Reference

Quick Glossary

Open this anytime, from any Act. Looking it up never affects your progress or locks.

Git

Saves versions of your project. Works offline.

GitHub

The website where Git projects are shared with a team.

Repository

A project folder Git is watching. Also called a "repo."

Codespaces

A ready-made computer in your browser. Nothing to install.

The terminal

Where you type commands and press Enter to run them.

Staging area

Where you collect exactly what you want to save next, using git add.

Commit

A permanent save point in Git's history. Made with git commit.

Fork

Your own personal copy of someone else's repository on GitHub.

Branch

A separate copy of the timeline, a safe space to try something without touching main.

Merge

Folding one branch's changes into another. Done with git merge.

.gitignore

Tells Git to never track certain files, but only from now on, not the past.

API key

A private password proving your program is allowed to use a service.

Rotate (a key)

Replacing an exposed credential with a new one. The real fix after a leak.
